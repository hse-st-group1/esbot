# Retrospective on the Review
## What worked well? 
- The reviewed team was willing to explain their code and reasoning to us, which helped us understand their thought process and reasoning.
- The provided template for the review was also helpful. It helped us set clear goals for the review.
- Communication with the other team was fast and productive. 

## What was difficult?
- Analysing the code of a project, which uses frameworks we have no experience with.
- The template contained some suggestions that we choose to leave out, given that the time needed to fill them out was not in a positive relation to the benefit it would have provided. 

## Are formal reviews a suitable method for your team? For which artefacts (spec, design, code) would you use them again?
- We chose an informal review type (=Walkthrough), given that none of us have previous experience with the framework used and the programming language C#. This allowed us to save time and review more efficiently, given that we could ask the reviewed Team whether something was a conscious code choice or just C# style. It also made it more efficient to analyse the List of Findings, given that they could be discussed directly. 
For code analysis we would choose this method again. 
- For review of documentation artefacts we choose peer review as a method. This worked well. We could review at our own pace. And given that we expected to find little issues and none of them needed to be fixed, a discussion with the other team was not necessary. 
## One concrete improvement you would make for the next review round.
- Plan a longer time frame for the walkthrough meeting. 
- Formulate questions more precisely for clearer communication.
